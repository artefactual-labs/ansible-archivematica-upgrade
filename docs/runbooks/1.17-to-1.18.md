# Archivematica 1.17 to 1.18 Upgrade Sample Runbook

This sample runbook describes one way to upgrade a single-host Archivematica
1.17.x installation to Archivematica 1.18.x, Storage Service 0.24.x, and
Elasticsearch 8 with the `artefactual.archivematica_upgrade` role.

This workflow currently supports single-host installations managed with
`ansible-archivematica-src`. The role consumes the existing
`archivematica_src_*` installer variables for database names, Storage Service
database mode, service configuration, and version checks.

The role runs one explicit phase per invocation. The
`ansible-archivematica-src` installer invocation remains outside the role and
must run only after the role reaches the installer handoff step.

Adapt this runbook to your deployment repository, inventory, maintenance
process, and rollback process before using it in production.

## Phase Runner Playbook

The commands below assume your deployment repository contains this phase
runner at `playbooks/awx.am-upgrade.yml`:

```yaml
---
- name: Run one Archivematica upgrade phase
  hosts: "{{ target }}"
  become: true
  roles:
    - role: "artefactual.archivematica_upgrade"
```

If your deployment repository does not already have one, copy the sample from
`examples/playbooks/awx.am-upgrade.yml` in this role. The `target` variable is
required because the play uses it as the host pattern.

Install the role as `artefactual.archivematica_upgrade` through your
deployment repository's Ansible requirements before running the phase runner.

## Preparation

Run commands from your deployment repository root. The shell variables below
are not required by the role; they keep this sample runbook concise and help
operators reuse the same values across phases:

```bash
TARGET_HOST=archivematica.example.org
UPGRADE_ID="1.18-upgrade"
UPGRADE_ROOT="/var/lib/archivematica-upgrades"
UPGRADE_PLAYBOOK="playbooks/awx.am-upgrade.yml"
```

Use the same `UPGRADE_ID` for every phase in one upgrade run so the role can
reuse the state and checkpoints it stores under `$UPGRADE_ROOT/$UPGRADE_ID`.
The sample uses `1.18-upgrade` because a production host normally has one
active 1.18 upgrade. Use a different ID for rehearsals, abandoned runs, or
multiple retained upgrade state directories on the same host.

`UPGRADE_ROOT` is the persistent state and backup root on the target host. Use
a filesystem with enough space for SQL dumps, configuration archives,
Elasticsearch filesystem backups, and migration working data.

Before starting any host, confirm:

- A recoverable external backup or snapshot exists for the target host.
- MySQL root client access works on the host.
- Elasticsearch 6 data is under `archivematica_upgrade_es_data_dir`, usually
  `elasticsearch_data_dir` or `/var/lib/elasticsearch`.
- Dashboard and Storage Service API credentials are valid in inventory.
- The host can reach `packages.archivematica.org` for Archivematica packages
  and `artifacts.elastic.co` for the temporary Elasticsearch 6 archive and
  checksum. For restricted-network hosts, configure the deployment's internal
  package source and set `archivematica_upgrade_es6_archive_url` with a
  matching `archivematica_upgrade_es6_checksum`.
- The target host has an existing non-root local account that can run the
  temporary Elasticsearch 6 process. By default, the role uses the Ansible SSH
  user; set `archivematica_upgrade_es6_user` when connecting as `root` or when
  another account should run the temporary process.
- `archivematica_upgrade_application_services` and
  `archivematica_upgrade_ignored_common_local_services` match the deployment.
  Include every local unit that can submit work, expose Archivematica, or
  depend on Elasticsearch during cutover.
- There are no active transfers, waiting MCP jobs, queued watched-directory
  contents, or configured Elasticsearch snapshot repositories.
- The target host already receives `ansible-archivematica-src` installer
  variables for Archivematica 1.18, Storage Service 0.24, Elasticsearch 8, full
  search indexing, and the required `archivematica_src_*` settings.
- The `ansible-archivematica-src` installer has not yet been run for the 1.18
  target state.

## Readiness Check

This phase is non-disruptive. It verifies the external recovery backup safety
gate, target installer variables, Elasticsearch 6 source state, idle pipeline
state, database collation, and storage capacity.

If this phase fails, stop and resolve the reported condition. Re-run
`check-readiness` with the same `UPGRADE_ID` until it passes. Do not run
`prepare` or the normal installer while readiness is failing.

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_snapshot_confirmed=true" \
  -e "archivematica_upgrade_phase=check-readiness" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

## Prepare

Run this phase only after scheduled downtime has started and Archivematica is
not expected to remain available. `prepare` repeats readiness and idle checks,
refreshes the Archivematica apt repository key, records source AIP and
application state, stops Archivematica services, captures Elasticsearch 6
index counts, stops Elasticsearch, and creates SQL, configuration,
Elasticsearch filesystem, and backup-manifest artifacts under
`$UPGRADE_ROOT/$UPGRADE_ID`.

The source state is kept under `$UPGRADE_ROOT/$UPGRADE_ID/state/`.
Elasticsearch 6-to-8 migration state is kept separately under
`$UPGRADE_ROOT/$UPGRADE_ID/state/migrations/elasticsearch-6-to-8/`.

If this phase fails, keep the maintenance window active and inspect the
reported task failure before proceeding. Fix the underlying issue and re-run
`prepare` with the same `UPGRADE_ID`; the role reuses completed checkpoints and
backup artifacts where safe. Do not run `cutover-before-install` until
`prepare` completes successfully.

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_snapshot_confirmed=true" \
  -e "archivematica_upgrade_phase=prepare" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

## Cutover Before Install

This phase hands off from the old packaged Elasticsearch 6 service to the
normal 1.18 installer. It keeps Archivematica services stopped, refreshes the
Archivematica apt repository key, removes the packaged Elasticsearch 6 service
and package, and preserves the old Elasticsearch 6 configuration, data, and log
directories with `-before-upgrade` suffixes.

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_phase=cutover-before-install" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

## Installer Handoff

Run your `ansible-archivematica-src` installation automation now, configured
for Archivematica 1.18, Storage Service 0.24, Elasticsearch 8, and full search
indexing. Do not run unrelated host provisioning unless your deployment
requires it for the handoff. A common tag-limited handoff looks like this:

```bash
ansible-playbook playbooks/awx.am-install.yml \
  --tags "elasticsearch,archivematica-src" \
  -l "$TARGET_HOST" \
  -e "target=$TARGET_HOST"
```

## Finish Upgrade Phases

Run the remaining phases in order.

`cutover-after-install` confirms the installer put Elasticsearch 8 in place and
keeps Archivematica services stopped for migration.

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_phase=cutover-after-install" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

`migrate` builds a temporary Elasticsearch 6 node from the preserved source
data, copies the Elasticsearch 6 indices into Elasticsearch 8 with remote
reindexing, stores asynchronous task IDs for resume, restores the
Elasticsearch 8 reindex configuration, and stops the temporary Elasticsearch 6
node.

If this phase fails, inspect the task output before retrying. Common causes are
Java 11 installation problems, Elasticsearch archive download issues,
temporary Elasticsearch startup failures, non-empty destination indices, or
failed remote-reindex tasks.

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_phase=migrate" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

`validate` starts services, checks Dashboard and Storage Service endpoints and
APIs, verifies Elasticsearch 8 index counts, captures destination AIP and
application state, and compares it with the source state recorded by
`prepare`.

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_phase=validate" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

## Rollback from Role Backups

Use this section only after `prepare` has completed and rollback artifacts
under `$UPGRADE_ROOT/$UPGRADE_ID/backups/` are still present.

The role restores application databases, Dashboard processing configuration
files, and, for this workflow, preserved Elasticsearch 6 directories to the
state captured during `prepare`. The `/etc` and Elasticsearch filesystem
archives are retained as manual safety artifacts; the role does not
automatically unpack them during restore.

The role does not reinstall or downgrade Elasticsearch. If
`cutover-before-install` or the installer handoff has already run, first return
the target host to its Archivematica 1.17 inventory and installer variables.
Then run your `ansible-archivematica-src` deployment automation so
Archivematica 1.17, Storage Service 0.23, and packaged Elasticsearch 6 are
installed again. A common tag-limited source-version handoff looks like this:

```bash
ansible-playbook playbooks/awx.am-install.yml \
  --tags "elasticsearch,archivematica-src" \
  -l "$TARGET_HOST" \
  -e "target=$TARGET_HOST"
```

Confirm the packaged `elasticsearch` package is installed at version 6.x before
continuing. It does not need to be running; `restore-backup` stops it before
moving Elasticsearch directories.

Rollback path by phase:

- Before `prepare` completes: no role backup set exists yet. Use the external
  recovery backup if a full host rollback is required.
- After `prepare`, before `cutover-before-install`: packaged Elasticsearch 6 is
  still installed but may be stopped. Run `restore-backup-check`, then
  `restore-backup` if returning to the prepared backup state.
- After `cutover-before-install` or installer handoff: restore the 1.17
  inventory and installer variables, run the source-version installer handoff,
  then run `restore-backup-check` and `restore-backup`.
- After rollback data removal: role-level restore is no longer available. Use
  the external recovery backup if needed.

First validate the rollback artifacts. This phase is non-destructive:

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_phase=restore-backup-check" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

After the check passes and packaged Elasticsearch 6 is installed, run the
confirmed restore:

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_restore_confirmed=true" \
  -e "archivematica_upgrade_phase=restore-backup" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

`restore-backup` validates the backup artifacts again, stops local
Archivematica and Elasticsearch services, stops the temporary Elasticsearch 6
node, moves preserved Elasticsearch 6 directories back into place, restores the
MCP and Storage Service databases, and restores Dashboard processing
configuration files. It records a `restore-started` checkpoint and leaves
service startup and 1.17 application validation to the operator or normal
deployment automation.

## Cleanup

Run cleanup only after `validate` passes and operator acceptance checks are
complete. This removes temporary Elasticsearch 6 runtime and download files but
keeps rollback backups.

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_cleanup_confirmed=true" \
  -e "archivematica_upgrade_phase=cleanup" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```

After cleanup, the captured upgrade state, checkpoints, and backups remain
under `$UPGRADE_ROOT/$UPGRADE_ID/`. The preserved packaged Elasticsearch 6
configuration, data, and logs also remain under their `-before-upgrade` paths
for rollback or inspection.

Once the rollback retention period has ended and the upgraded system has been
accepted, remove rollback data with a second cleanup run:

```bash
ansible-playbook "$UPGRADE_PLAYBOOK" \
  -l "$TARGET_HOST" \
  -e "archivematica_upgrade_cleanup_confirmed=true" \
  -e "archivematica_upgrade_remove_rollback_data_confirmed=true" \
  -e "archivematica_upgrade_phase=cleanup" \
  -e "archivematica_upgrade_id=$UPGRADE_ID" \
  -e "archivematica_upgrade_root=$UPGRADE_ROOT" \
  -e "target=$TARGET_HOST"
```
